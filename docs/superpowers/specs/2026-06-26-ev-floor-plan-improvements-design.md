## 1. 目的とゴール

EV充電設備の平面図・配線ルート図を補助金要件に沿って作図するツールに対し、以下7項目を実装する。

1. AI読取（#1）: ラフ図PDFをClaude(vision)で解析し、充電スペース・充電器・寸法を**実図形として自動配置**
2. 保存・復元（#2）: プロジェクトの保存/読込と自動保存（作業消失の防止）
3. Undo/Redo（#3）: 堅牢な履歴管理（現状の壊れた `undo()` を置換）
4. prompt()廃止（#4）: ブロッキングな `prompt()`/`alert()` をプロパティフォームUIへ
5. 要件検証＋警告（#5）: 補助金要件の**記載漏れ検査**と不足の差分表示
6. 自動読取レビューUI（#6）: 抽出結果を「プレビュー→採用/却下→確定」で反映
7. 大ファイル分割＋テスト（#7）: 肥大ファイルの責務分割とユニットテスト導入

### 成功基準
- ブラウザを閉じても作業が失われない（自動保存＋手動保存）。
- 任意の作図操作を Ctrl+Z / Ctrl+Y で巻き戻し・やり直しできる。
- `prompt()` による入力が全廃され、すべて右パネルのフォームで編集できる。
- 補助金要件の必須寸法・注釈の不足が画面上で具体的に分かる。
- ラフ図PDFから、最低限「充電スペースの台数・並び」「充電器位置」「主要寸法」が図形として自動配置され、ユーザーがレビューして確定できる。
- `node --test` で純ロジック（座標変換・シリアライズ往復・検証ルール・縮尺パース）のテストが緑になる。

### 非ゴール / 将来課題
- 注釈を純粋なデータモデルに全面移行する大規模リファクタ（後述 B案）は本スコープ外。
- DWG対応、クラウド保存、複数ユーザー同時編集は対象外。
- AIによる配線ルート図の自動生成は対象外（今回は平面図のラフ図読取に限定）。

## 3. 横断アーキテクチャ — 状態シリアライズ（採用: A案）

### 決定
注釈はSVG DOM要素（`data-type` / `data-id` / ジオメトリを持つ `<g>`）として存在する現構造を維持し、**DOM ⇄ JSON を相互変換する `state-serializer.js` を新設**する（A案）。保存(#2)・Undo/Redo(#3)・要件検証(#5)の全機能がこの単一モジュールを共有する。

### 却下した代替案（B案）
注釈を純粋なデータ配列で保持しSVGをそこから描画する全面リファクタ。理想的だが全ツールの書き換えを伴い高リスク・高工数。本スコープでは採用しない（将来課題）。

### `state-serializer.js` インターフェース
```
StateSerializer.serialize() -> ProjectState        // 現在の図面状態をJSON化
StateSerializer.deserialize(ProjectState) -> void   // JSONからSVGを復元
StateSerializer.snapshot() -> string                // 履歴用の軽量スナップショット
StateSerializer.restore(string) -> void             // スナップショットから復元
```

### ProjectState スキーマ（保存ファイル形式）
```jsonc
{
  "version": 1,
  "savedAt": "2026-06-26T12:00:00+09:00",
  "titleBlock": { "siteName": "...", "projectName": "...", "drawingName": "...",
                  "author": "...", "scale": "A3:1/100", "date": "...", "northAngle": 0 },
  "dxf": { "fileName": "....dxf", "loaded": true },   // ジオメトリ実体は再読込前提（後述）
  "annotations": [
    { "type": "charging-space", "id": "a1", "x": 0, "y": 0, "w": 2.5, "h": 5.0, "label": "" },
    { "type": "charger", "id": "a2", "x": 1.2, "y": 0.3, "rotation": 0, "label": "①" },
    { "type": "dimension", "id": "a3", "x1": 0, "y1": 0, "x2": 2.5, "y2": 0, "value": "2500", "color": "#0066cc" }
    // ... road-marking / wheel-stop / bollard / lighting / foundation / text /
    //     leader(wire/equipment/conduit) / wiring-route / cubicle / pole / handhole / pullbox / existing-charger / boundary-rect
  ],
  "viewBox": "-100 -100 200 200"
}
```

#### DXF/PDF参照の扱い
- DXFジオメトリは大きい（テスト用 約800KB）ため保存JSONには**含めない**。保存時はファイル名のみ記録し、復元時は「同じDXFを再読込してください」と促す（参照消失時もアノテーションは復元可能）。
- 注釈座標はDXFと同じワールド座標系で持つため、DXFを再読込すれば整合する。
- PDF参照も実体は保存せずファイル名のみ記録（再読込導線を表示）。

#### 各注釈タイプの直列化責務
- `svg-engine.js` の各 `createXxx` が生成する `<g>` に、復元に必要な属性（型・id・座標・寸法・ラベル・色・回転）を**漏れなく `data-*` として持たせる**ことを前提とする。不足している型は属性付与を補う（フェーズ1の付随作業）。

### エラーハンドリング
- 不明な `type` や壊れたレコードはスキップしつつ警告ログ＋トーストで件数表示（全体失敗にしない）。
- `version` 不一致は将来のマイグレーション余地として検知のみ（現状は v1 のみ）。

## 4. フェーズ1 — 土台（#3 Undo/Redo, #2 保存・復元）

### 4.1 履歴管理 `history.js`
- 方式: スナップショットスタック（`StateSerializer.snapshot()` の文字列を積む）。
- API: `History.push()` / `History.undo()` / `History.redo()` / `History.canUndo()` / `History.canRedo()`。
- 記録タイミング: 注釈の追加・削除・移動・プロパティ変更の確定時に `push()`。
- 上限: 直近 N=50 件（メモリ保護）。
- キーバインド: `Ctrl+Z` = undo, `Ctrl+Y` / `Ctrl+Shift+Z` = redo。
- 既存 `app.js:undo()`（outerHTMLで末尾削除）は撤去し本モジュールに置換。
- UI: ヘッダーに Undo/Redo ボタンを追加し `canUndo/canRedo` で活性制御。

### 4.2 保存・復元
- **手動保存**: ヘッダーに「保存」ボタン。`StateSerializer.serialize()` → `.json` をダウンロード（ファイル名は設置場所名 + 日付）。
- **読込**: 「開く」ボタンで `.json` を選択 → `deserialize()`。
- **自動保存**: 変更のたび（debounce 1s）に `localStorage` へ保存。起動時に未保存データがあれば「前回の作業を復元しますか？」を提示。
- **エラーハンドリング**: localStorage容量超過/パース失敗時はトースト警告し、手動保存を促す。

### 4.3 テスト（フェーズ1）
- `state-serializer`: 代表的な全注釈タイプを配置 → serialize → deserialize → 再serialize が一致（往復テスト）。
- `history`: push/undo/redo の状態遷移、上限トリミング。

## 5. フェーズ2 — UX（#4 prompt()廃止, #6 レビューUI）

### 5.1 prompt()廃止
対象（`app.js` 内の `prompt()` 使用箇所）:
- 囲み線(boundary-rect)、分電盤(cubicle)、建柱(pole)、プルボックス(pullbox)、既設充電器(existing-charger)、基礎(foundation 既定値)。
- 方式: ツールはクリック地点に**既定値で即配置** → 右パネルのプロパティフォームで編集（既存の properties-section を各型用フォームに拡張）。
- バリデーション: 数値項目は型・正値チェック。不正値はフィールド単位でエラー表示。
- `alert()` も非ブロッキングなトースト/インライン表示へ統一。

### 5.2 自動読取レビューUI（#6）
- `pdf-auto-reader.js` の出力を**即時にSVGへ反映しない**よう変更。抽出結果を候補リスト（type / text / 座標 / 分類）としてモーダルに表示。
- 操作: 各候補を「採用/却下」、分類のドロップダウン変更、一括採用。
- 「確定」で採用分のみ生成 → `History.push()`。
- これはフェーズ4のAI結果レビューと**同じUIコンポーネント**（`review-panel.js`）を使う前提で設計する。

## 6. フェーズ3 — 検証＋整理（#5 要件検証, #7 分割＋テスト）

### 6.1 要件検証 `requirement-validator.js`
補助金要件は「数値下限」より「**必須の寸法・注釈の記載漏れ**」検査が本質（ルールブックで確認済み）。以下を**ルール定義テーブル**として持ち、現状の `updateChecklist()`（型の有無のみ）を拡張する。

平面図の検査例:
- ①図面基本情報: タイトルブロック各項目（設置場所/縮尺/作成者/図面名称/作成日）が埋まっているか。
- ②充電スペース: スペースに**幅・奥行きの寸法線**が紐づいているか。
- ③充電設備位置: スペースと充電器の**位置関係寸法**があるか。
- ④基礎: 基礎に**寸法（たて・よこ・高さ）**が記載されているか。
- ⑥路面表示: 路面表示の**位置寸法**があるか。
- ⑨防護用部材 / ⑩車止め: それぞれ充電設備との**寸法**があるか。
- ⑪電灯: 設置位置があるか。

判定結果は右パネルのチェックリストを ○（必須・未）/ ✔（充足）/ ⚠（あるが寸法不足）/ −（非該当）で表現し、不足項目は「⚠ 基礎の寸法が未記載」のように**具体的な不足内容**を表示する。

> 注: 「寸法が紐づいているか」は近接判定（注釈と寸法線の座標距離）で近似する。厳密な関連付けが難しい項目は「寸法線が1本以上存在するか」等の緩い判定にフォールバックし、過検出より見落とし防止を優先する。判定ロジックは純関数として切り出しテストする。

## 7. フェーズ4 — AI読取（#1）

### 7.1 backend
- 新規 `server.js` を追加し、既存 `serve.js` の静的配信機能を取り込んだ上で `POST /api/analyze-sketch` を実装する。`.claude/launch.json` の起動コマンドを `server.js` に切り替える。`serve.js` は後方互換のため残置（最終的に削除可否は実装時判断）。
- 静的配信に加え `POST /api/analyze-sketch` を実装。
- `ANTHROPIC_API_KEY` は**サーバ側の環境変数**から読む（ブラウザに露出させない）。
- リクエスト: ラフ図の**ページ画像（PNG/base64）** + DXFの寸法情報（スケール手がかり）。
- レスポンス: 後述の構造化JSON。
- エラー: APIキー未設定/レート制限/失敗時は明確なエラーをJSONで返し、フロントはトースト表示。

### 7.2 Claude vision 呼び出し
- Anthropic Messages API（vision）。画像 + 指示プロンプトで構造化JSONを要求（JSONスキーマを明示、`tool_use`/JSON modeで安定化）。
- モデル: コスト/精度バランスから既定は Sonnet系。精度不足時に Opus へ切替可能な設定にする。モデルIDは claude-api スキルで実装時に最終確認する。
- 出力スキーマ（例）:
```jsonc
{
  "charging_spaces": [ { "count": 6, "layout": "horizontal", "width_mm": 2500, "depth_mm": 5000 } ],
  "chargers": [ { "label": "①", "near_space_index": 0 } ],
  "dimensions": [ { "value_mm": 900, "orientation": "horizontal" } ],
  "road_markings": [ { "near_space_index": 0 } ],
  "notes": ["..."]
}
```
※ 位置はラフ図の相対配置 + DXF基準で確定する。座標確定が困難な要素は寸法・台数の情報のみ採用。

### 7.3 図形変換と配置
- 返却JSONを実図形（スペース枠・充電器・寸法・路面表示）へ変換する `sketch-to-shapes.js`。
- 生成結果はフェーズ2の**レビューUI（`review-panel.js`）**に流し、ユーザー確定後に `History.push()`。
- 現状の正規表現分類（`_classifyAndCreate`）は **AIが使えない/失敗時のフォールバック**として残す。

### 7.4 テスト（フェーズ4）
- `sketch-to-shapes`: サンプルJSON → 期待図形数・座標の純ロジックを単体テスト。
- backendはAPIキー無し環境向けにモック応答での結合確認を用意。

## 9. リスクと緩和
- **シリアライズの属性漏れ**: 既存 `createXxx` が復元に必要な属性を全て持たない可能性 → フェーズ1で型ごとに往復テストし不足属性を補完。
- **要件検証の関連付け精度**: 寸法と対象の紐付けは近接近似 → 見落とし防止優先の緩い判定 + テストで調整。
- **AIの座標精度**: ラフ図からの絶対座標確定は困難 → 台数/寸法/相対配置を主情報とし、最終位置はユーザーがレビューUIで調整。
- **APIコスト**: vision呼び出しは課金 → 既定Sonnet、画像解像度を必要十分に圧縮、呼び出しはユーザー明示操作時のみ。
- **大ファイル分割の回帰**: 触る範囲限定 + 分割後に手動動作確認とテスト。
